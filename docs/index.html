Looped nanochat

A two-pass, shared-weight routing-loop Transformer — run the trunk twice, and let each second-pass layer read an attention prior from the first pass. A controlled study of whether "looping" a model helps at fixed scale. At d24 (~1.38 B params), it wins or ties every eval.

How the routing loop works

Standard nanochat is a single-pass causal Transformer. The looped variant runs the same weights twice per forward.

  1. Pass 1 — raw. Run the ordinary causal Transformer. At three source layers, keep the normalized, RoPE-applied query/key features (q₁, k₁).
  2. Pass 2 — routed. Re-run the trunk from the token embeddings. Every layer adds a causal attention-logit prior built from its assigned source layer's (q₁, k₁), scaled by a learned per-layer gate.

How the bias enters one attention layer

Inside a pass-2 layer, the desired attention logits are its own scores plus a gated copy of the source layer's scores:

attn_logits  =  q₂·k₂ᵀ / √D   +   g · ( q₁·k₁ᵀ / √D )   g = softplus(gate_logit), one scalar per layer

Rather than form either N×N score matrix, the layer concatenates along the head dimension and runs one ordinary attention at width 2D — the dot product reproduces the sum exactly, and the gate is folded into the routing keys:

Reproduce

bash runs/speedrun_baseline.sh   # stock nanochat,  tag: baseline-d24
bash runs/run_looped.sh          # routing loop on, tag: looped-d24

# full generative eval (either tag)
torchrun --standalone --nproc_per_node=4 -m scripts.chat_eval -- \
    -i sft -g looped-d24 -a "GSM8K|HumanEval"

Looped flags for scripts.base_train: --routing-loop --routing-pattern=progressive --routing-gate-init=0.05. The routing config is stored in each checkpoint, so SFT / eval / inference rebuild the two-pass model automatically.
